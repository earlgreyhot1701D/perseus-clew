By scanning, you confirm you have the right to test this URL. AgentisLux reads raw HTML. It does not execute JavaScript. Scans are ephemeral and not stored.

What happens when you scan

Three steps. One report.

  1. Fetch and parse We fetch your HTML (no JavaScript execution). If an OpenAPI spec is auto-discovered, we parse it alongside.
  2. Layer 1 — deterministic scan Six frontend categories. Six API categories if a spec is present. Pattern matching, no AI. Same input, same score, every time.
  3. Layer 2 — agent simulation Three tasks attempted against your site as an agent would. Claude Haiku reports what it could and could not do. Linked back to Layer 1 findings.

The detailed findings.

SEM-001

Primary CTA is a styled div, not a button element

An agent landing on this page cannot confirm the checkout action is a button. The element has a click handler but uses a generic div tag, making it indistinguishable from static content in the DOM.

High
FORM-001

Phone input relies on placeholder for guidance

The phone field uses a placeholder attribute as its only indicator of expected format. An agent reading the form cannot determine the required format once a value is entered, because placeholders disappear on focus.

Medium
SD-001

Page missing JSON-LD structured data

No JSON-LD script tag is present on the page. An agent cannot use Schema.org structured data to confirm the page type, product details, or business identity. This makes the page harder to categorize without interpreting visible text.

Medium
NAV-002

Breadcrumb nav uses custom list markup

The breadcrumb navigation uses div elements with visual separators rather than a semantic ol or nav with aria-label. An agent parsing navigation hierarchy must infer the relationship from visual cues that are not present in raw HTML.

Low

What the agent experienced

I could see a cream-colored region containing the text "Continue to checkout."
